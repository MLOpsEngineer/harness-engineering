# SEC-bench: Automated Benchmarking of LLM Agents on Real-World Software Security Tasks

## 한줄 요약
실제 보안 엔지니어링 작업을 닮은 환경에서 LLM code agent를 자동 평가하기 위한 벤치마크 프레임워크를 제안한 논문으로, 보안 분야 eval harness를 어떻게 구성할지를 정면에서 다룬다.

## 문제 설정
기존 security benchmark는 synthetic challenge나 단순화된 vulnerability dataset에 많이 의존해 실제 보안 엔지니어가 다루는 복잡성과 모호성을 충분히 반영하지 못했다.

## 1. 논문이 먼저 세우는 문제
- 논문은 security benchmarking을 일반 코딩 benchmark의 하위 문제로 보지 않는다. 실제 보안 작업은 취약점 재현, 환경 일치, PoC 검증, patch 검증이 동시에 필요하므로 훨씬 더 까다롭다고 본다.
- 기존 보안 benchmark의 핵심 한계로 `verifiable high-quality proof-of-concept 입력` 부재를 든다. 논문은 기존 데이터셋의 label 품질 문제, web framework 편향, 수동 제작 challenge의 확장성 한계를 구체적으로 비판한다.
- 목표는 세 가지다. `검증된 PoC와 정확한 trigger condition을 갖는 high-quality vulnerability`, `자동 구축`, `현실적 security engineering workflow 반영`이다.

## 2. SEC-bench 프레임워크 구성
- SEC-bench는 `preprocessor`, `verifier`, `evaluator` 세 모듈로 구성된다.
- preprocessor는 public CVE database에서 메타데이터를 모으고 bug report를 수집한 뒤, 검증 가능한 interactive environment를 Docker로 만든다.
- verifier는 `SECVerifier`라는 specialized multi-agent scaffold로 취약점을 실제로 재현하고 검증한다.
- evaluator는 검증된 인스턴스를 바탕으로 Docker image와 artifact를 묶어, agent가 실제 security task를 수행하도록 benchmark task로 패키징한다.

## 3. 원문에서 보이는 데이터 구축 파이프라인
- 대상은 memory safety sanitizer로 검증 가능한 open-source C/C++ 프로젝트다.
- Step 1 메타데이터 수집에서 OSV database를 사용해 `38,201`개 잠재 인스턴스를 모으고, `7,926`개 프로젝트를 포괄한다.
- Step 2에서는 GitHub Issues, RedHat Bugzilla, Chromium tracker 등 이질적인 bug report source를 긁어오고 OSS-Fuzz 설정을 재사용해 `4,836`개 문서화된 인스턴스를 남긴다.
- Step 3 환경 구성에서는 project-specific Docker configuration과 agent-friendly harness를 붙여 sanitizer report가 가능한 `898`개 candidate를 남긴다.
- 최종적으로 verifier를 거쳐 `200`개 실제 CVE 인스턴스를 검증했다고 밝힌다.

## 4. 검증 오라클과 task formulation
- 이 논문에서 핵심 오라클은 memory safety sanitizer다.
- sanitizer report가 예상대로 발생하면 PoC를 accept하고, patch 이후 해당 report가 사라지면 fix를 valid하다고 본다.
- 즉 benchmark의 정답은 문자열 매칭이 아니라 `instrumented execution 결과`다.
- 이 설계는 보안 영역에서 deterministic oracle을 확보하기 위한 선택이며, DARPA AIxCC와도 결을 같이한다고 논문은 설명한다.
- evaluator는 이렇게 검증된 인스턴스를 두 가지 task로 바꾼다: `PoC generation`과 `vulnerability patching`.

## 5. 핵심 결과
- SEC-bench는 single-agent scaffold인 CodeAct 대비 `85.7%` 더 많은 실제 CVE 인스턴스를 검증했다고 주장한다.
- SOTA code agent 평가에서는 complete dataset 기준 최고 성능이 PoC generation `18.0%`, vulnerability patching `34.0%`다.
- 논문이 전달하는 메시지는 명확하다. real-world security task는 현재 code agent에게 여전히 매우 어렵고, 실용성까지 가려면 agent가 훨씬 더 practical, intelligent, autonomous해야 한다.

## 6. 원문 수준에서 읽히는 의미
- SEC-bench는 단순 데이터셋이 아니라 `benchmark construction harness` 자체가 논문의 기여다.
- 특히 bug report normalization, Dockerized environment setup, sanitizer-based oracle, gold patch packaging이 하나의 평가 시스템으로 묶여 있다는 점이 중요하다.
- 보안 영역에서는 eval harness가 production-like environment reconstruction까지 포함해야만 의미 있다는 점을 잘 보여 준다.

## Harness Engineering 관점
- harness engineering 관점에서 이 논문은 `좋은 agent를 만들려면 먼저 좋은 평가 환경이 필요하다`는 사실을 보안 영역에서 보여 준다.
- 특히 여기서 harness는 단순 채점 스크립트가 아니라, repository construction, isolated reproduction, gold patch generation, reproducible artifact packaging을 함께 묶는 평가 인프라다.
- 운영 harness와 eval harness가 분리된다는 점도 중요하다. production agent를 잘 굴리는 메커니즘과, 그것을 엄격하게 검증하는 메커니즘은 별도 설계 대상이다.

## 한계와 주의점
- 직접적으로 harness engineering을 개념화하는 논문은 아니다.
- 현재 직접 확인한 본문 범위는 주로 서론과 프레임워크 초반부다. verifier 내부 multi-agent 로직, evaluator 세부 scoring, baseline별 상세 failure taxonomy는 추가 독해가 필요하다.
- benchmark 논문이므로 production harness를 어떻게 설계할지보다 evaluation harness를 어떻게 만들지에 초점이 있다.
