# CEDAR: Context Engineering for Agentic Data Science

- 구분: 논문
- 발행일: 2026-01-10
- 저자: Rishiraj Saha Roy 외 3명
- 출처: arXiv
- 원문: https://arxiv.org/abs/2601.06606
- 관련성: 인접 연구

## 한줄 요약
데이터사이언스 태스크를 위해 structured prompt, 3-agent orchestration, interleaved plan/code generation, local execution, history rendering을 결합한 context engineering 시스템을 제안한 논문이다.

## 문제 설정
데이터사이언스 작업은 일반 coding agent보다 context restriction의 영향을 훨씬 크게 받는다. **데이터 크기** (수 GB의 CSV/DB를 통째로 context에 넣을 수 없음), **계산 제약** (무거운 연산은 LLM이 직접 할 수 없음), **privacy** (민감 데이터를 외부 API에 보낼 수 없음), **multi-step workflow** (EDA → 전처리 → 모델링 → 해석의 긴 파이프라인) 때문이다.

## 제안 방법

### Structured Prompt
- DS-specific input field를 가진 **structured prompt**를 출발점으로 삼는다.
- Task description, data description, file location, target metric, inputs/outputs 같은 필드를 **구조화해 입력**한다.
- 자유 형식 자연어보다 structured field가 agent의 작업 이해도를 높인다.

### 3-Agent Orchestration
- 시스템은 `orchestrator`, `text generator`, `code generator`의 **3-agent** 구조다.
- **Orchestrator**: `request_text`, `request_code`, `finish` 같은 tool call을 사용해 하위 agent를 부른다. 이 덕분에 free-form parsing 오류를 줄이고 role을 명확히 분리한다.
- **Text Generator**: 분석 계획, 해석, 문서화를 담당. 사람이 읽을 수 있는 설명을 생성한다.
- **Code Generator**: 실제 실행 가능한 Python 코드를 생성. 데이터 로딩, 전처리, 시각화, 모델링 등.
- 출력은 한 번에 최종 답을 뱉는 대신, 번호가 매겨진 **interleaved text/code 블록**으로 구성된 notebook-like workflow다.

### Data Locality
- 코드 실행은 **Docker 기반 로컬 환경**에서 일어나며, 데이터는 가능한 로컬에 남긴다.
- LLM에는 **aggregate statistics와 instruction만** 넣는다. raw data를 context에 올리지 않는다.
- 이는 privacy 보호와 context 효율성을 동시에 달성하는 전략이다.

### Smart History Rendering
- context 관리는 **smart history rendering**과 iterative code generation으로 처리한다.
- 이전 단계의 코드와 결과를 요약해 다음 단계의 context에 포함한다. 전체 history를 그대로 유지하지 않고, 현재 단계에 필요한 정보만 선별적으로 rendering한다.

## 결과와 시사점
- Kaggle challenge를 사례로 CEDAR의 viability를 보인다.
- 핵심 메시지: DS에서 LLM을 쓸 때 중요한 것은 모델 하나의 지능보다 **workflow를 사람이 읽을 수 있게 쪼개고, data locality를 유지하며, context를 계속 intelligible하게 유지하는 것**이다.
- 계획과 코드 생성의 역할 분리가 **transparency를 높이고**, notebook-like artifact를 남겨 **human inspection을 쉽게** 만든다.
- orchestrator의 tool call 기반 routing은 free-form text parsing보다 안정적이고 디버깅이 용이하다.

## Harness Engineering 관점
- 도메인이 달라도 핵심은 같다. **무엇을 context로 올리고 무엇을 외부 런타임에 남길지** 설계하는 것이 성능을 결정한다.
- CEDAR는 harness를 `structured prompt + orchestrator routing + local execution + history rendering`의 묶음으로 구현한다. 이는 코딩 agent의 harness와 구조적으로 동일하다.
- data locality 전략은 Anthropic의 context engineering에서 말하는 **"무엇을 뺄까가 더 중요하다"**의 구체적 적용이다. raw data를 빼고 summary만 넣는다.
- 3-agent orchestration은 Anthropic의 initializer/coding agent 분리와 같은 원리다. 역할을 분리하면 각 agent의 context가 깨끗해진다.

## 한계와 주의점
- 데이터사이언스 특화 구조라 일반 소프트웨어 작업으로 바로 확장하기는 어렵다.
- 논문 자체도 novelty보다 application/system report 성격이 강하다.
- 주로 context engineering과 transparency에 초점이 있고, 광범위한 benchmark 실험은 제한적이다.
- Kaggle challenge라는 비교적 well-defined task에서의 결과라, open-ended 분석 작업으로의 일반화는 추가 검증이 필요하다.
