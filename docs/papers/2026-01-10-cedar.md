# CEDAR: Context Engineering for Agentic Data Science

- 구분: 논문
- 발행일: 2026-01-10
- 저자: Rishiraj Saha Roy 외 3명
- 출처: arXiv
- 원문: https://arxiv.org/abs/2601.06606
- 관련성: 인접 연구

## 한줄 요약
데이터사이언스 태스크를 위해 structured prompt, 3-agent orchestration, interleaved plan/code generation, local execution, history rendering을 결합한 context engineering 시스템을 제안한 논문이다.

## 문제 설정
데이터사이언스 작업은 데이터 크기, 계산 제약, privacy, multi-step workflow 때문에 일반 coding agent보다 context restriction의 영향을 훨씬 크게 받는다.

## 제안 방법
- CEDAR는 DS-specific input field를 가진 `structured prompt`를 출발점으로 삼는다. Task description, data description, location, metric, inputs/outputs 같은 필드를 구조화해 입력한다.
- 시스템은 `orchestrator`, `text generator`, `code generator`의 `3-agent` 구조다.
- 출력은 한 번에 최종 답을 뱉는 대신, 번호가 매겨진 `interleaved text/code` 블록으로 구성된 notebook-like workflow다.
- orchestrator는 `request_text`, `request_code`, `finish` 같은 tool call을 사용해 하위 agent를 부른다. 이 덕분에 free-form parsing 오류를 줄이고 role을 명확히 분리한다.
- 코드 실행은 Docker 기반 로컬 환경에서 일어나며, 데이터는 가능한 로컬에 남기고 aggregate statistics와 instruction만 LLM에 넣는다.
- context 관리는 `smart history rendering`과 iterative code generation으로 처리한다.

## 결과와 시사점
- 논문은 canonical Kaggle challenge를 사례로 CEDAR의 viability를 보인다.
- 핵심 메시지는 DS에서 LLM을 쓸 때 중요한 것은 모델 하나의 지능보다 `workflow를 사람이 읽을 수 있게 쪼개고`, `data locality를 유지하며`, `context를 계속 intelligible하게 유지하는 것`이라는 점이다.
- 계획과 코드 생성의 역할 분리가 transparency를 높이고, notebook-like artifact를 남겨 human inspection을 쉽게 만든다.

## Harness Engineering 관점
- 도메인이 달라도 핵심은 같다. 무엇을 context로 올리고 무엇을 외부 런타임에 남길지 설계하는 것이 성능을 결정한다.
- CEDAR는 harness를 `structured prompt + orchestrator routing + local execution + history rendering`의 묶음으로 구현한다.

## 한계와 주의점
- 데이터사이언스 특화 구조라 일반 소프트웨어 작업으로 바로 확장하기는 어렵다.
- 논문 자체도 novelty보다 application/system report 성격이 강하다.
- 주로 context engineering과 transparency에 초점이 있고, 광범위한 benchmark 실험은 제한적이다.
