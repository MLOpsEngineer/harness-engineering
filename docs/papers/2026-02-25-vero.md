# VeRO: An Evaluation Harness for Agents to Optimize Agents

- 구분: 논문
- 발행일: 2026-02-25
- 저자: Varun Ursekar 외 4명
- 출처: arXiv
- 원문: https://arxiv.org/abs/2602.22480
- 관련성: 직접

## 한줄 요약
에이전트가 다른 에이전트를 edit-execute-evaluate loop로 개선하는 task를 reproducible하게 측정하기 위해 설계된 evaluation harness 연구로, `agent optimization` 자체를 하나의 coding task로 벤치마킹한다.

## 문제 설정
에이전트가 타깃 에이전트를 개선하는 작업(agent-optimizing-agent)은 점점 중요해지고 있지만, **stochastic LLM behavior와 deterministic code execution이 섞여** 있어 재현 가능한 평가가 어렵고 공통 benchmark도 부재했다. optimizer agent들을 공정하게 비교하려면 먼저 evaluation harness를 제대로 설계해야 한다.

## 제안 방법

### VERO의 의미
- VERO는 `Versioning, Rewards, and Observations`의 약자다.
- 이 세 가지가 agent optimization을 reproducible하게 만들기 위한 핵심 요구사항이다.

### Evaluation Harness 요구사항
- 논문은 target agent optimization을 정식화하고, 다음 요구사항을 명시한다:
  1. **Versioning**: optimizer가 만든 각 agent 버전을 구분하고 추적할 수 있어야 한다. Git worktree 기반 구현.
  2. **Budget Enforcement**: optimizer가 무한히 시도하지 못하도록 evaluation call 수를 제한한다. 기본 설정 `B = 8`.
  3. **Permission Control**: optimizer agent의 filesystem access를 통제해 의도하지 않은 변경을 방지한다.
  4. **Reproducible Execution**: 같은 agent 코드를 같은 데이터에 실행하면 같은 결과가 나와야 한다.
  5. **Structured Tracing**: optimizer의 모든 행동(코드 수정, 평가 호출, 결과 관찰)을 구조화해 기록한다.
  6. **Standardized Observation Interface**: optimizer가 평가 결과를 일관된 형식으로 받아볼 수 있어야 한다.

### 인프라 구현
- **Git Worktree**: 각 optimization 시도를 독립된 worktree에서 실행. commit 단위로 versioning하므로 이전 버전으로 쉽게 돌아갈 수 있다.
- **Dataset Abstraction**: 평가에 사용할 데이터를 표준화된 인터페이스로 제공.
- **Filesystem Access Control**: optimizer가 수정할 수 있는 파일 범위를 제한.
- **Experiment Database**: 모든 실험 결과를 구조화해 저장. 실험 간 비교와 분석을 지원.
- **Evaluator Abstraction**: 다양한 평가 방법(exact match, LLM judge, human eval)을 통합 인터페이스로 제공.

## 결과와 시사점

### Benchmark Tasks
- `MATH`, `GAIA`, `GPQA`, `SimpleQA`, `TAU-Bench Retail`로 구성된다.
- task 유형이 다양해 optimizer의 범용성을 측정할 수 있다.

### Task 특성에 따른 결과 차이
- **Reasoning-heavy task**(GPQA, MATH)는 개선 폭이 작았다. 모델의 추론 능력 자체가 병목이면 harness 최적화로 개선하기 어렵다.
- **Tool-use-heavy task**(GAIA, Retail, SimpleQA)는 일관된 lift가 나타났다. 도구 사용이 핵심인 task에서는 harness 최적화의 효과가 크다.
- 도구 사용형 task에서 대략 **8~9% 수준 lift**가 관찰된다.

### Optimizer 행동 분석
- tracing 분석 결과, current optimizer는 대개 **prompt modification에 과도하게 의존**하고 modification diversity가 낮다.
- 즉 optimizer가 prompt만 바꾸려 하고, tool definition이나 workflow 구조를 바꾸는 시도는 드물다. 이는 harness engineering의 optimization space가 아직 충분히 탐색되지 않았음을 시사한다.

### Variance-Performance Tradeoff
- optimizer instruction template에 따라 **variance-performance tradeoff**가 나타난다.
- 공격적인 instruction은 peak gain은 크지만 분산도 커지고 **cross-task regression**이 생길 수 있다.
- 보수적인 instruction은 안정적이지만 개선 폭이 작다.
- simpler target agent일수록 **optimization headroom**이 더 크다는 점도 case study에서 보인다.

## Harness Engineering 관점
- harness engineering이 agent를 개선하는 대상일 뿐 아니라, **개선 과정을 측정하는 harness도 필요**하다는 점을 잘 보여준다. meta-harness의 개념이다.
- agent optimizer를 비교하려면 **target agent보다 먼저 evaluation harness를 제대로 설계**해야 한다는 점이 논문의 핵심 기여다.
- versioned snapshot, structured trace, budgeted evaluator 같은 메커니즘은 **self-improving agent 연구의 기본 인프라**로 읽을 수 있다.
- "reasoning task에서는 harness 최적화 효과가 작고, tool-use task에서는 크다"는 결과는 harness engineering의 **적용 범위에 대한 현실적 기대치**를 제시한다.
- optimizer가 prompt만 바꾸려 한다는 관찰은, harness engineering의 optimization space(tool, middleware, workflow 등)를 agent가 아직 충분히 탐색하지 못하고 있음을 보여준다.

## 한계와 주의점
- 평가 프레임워크 논문이라 직접적인 product harness recipe는 제한적이다.
- budget을 evaluation call 수로만 정의해 **token/API cost variance**를 완전히 통제하지 못한다는 한계를 논문도 인정한다.
- target task는 유용하지만, production coding workflow 전체를 대변하진 않는다.
- optimizer의 행동 다양성 부족은 현재 LLM의 한계일 수 있으며, 모델 발전에 따라 결과가 달라질 수 있다.
