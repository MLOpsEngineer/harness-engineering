# VeRO: An Evaluation Harness for Agents to Optimize Agents

- 구분: 논문
- 발행일: 2026-02-25
- 저자: Varun Ursekar 외 4명
- 출처: arXiv
- 원문: https://arxiv.org/abs/2602.22480
- 관련성: 직접

## 한줄 요약
에이전트가 다른 에이전트를 edit-execute-evaluate loop로 개선하는 task를 reproducible하게 측정하기 위해 설계된 evaluation harness 연구로, `agent optimization` 자체를 하나의 coding task로 벤치마킹한다.

## 문제 설정
에이전트가 타깃 에이전트를 개선하는 작업은 increasingly 중요하지만, stochastic LLM behavior와 deterministic code execution이 섞여 있어 재현 가능한 평가가 어렵고 공통 benchmark도 부재했다.

## 제안 방법
- VERO는 `Versioning, Rewards, and Observations`의 약자다.
- 논문은 target agent optimization을 정식화하고, optimizer coding agent가 target agent code를 수정하도록 둔다.
- harness 요구사항도 명시한다: versioning, budget enforcement, permission control, reproducible execution, structured tracing, standardized observation interface.
- 이를 위해 Git worktree, Dataset abstraction, Filesystem access control, Experiment Database, Evaluator abstraction을 제공한다.
- 평가 도구는 commit 단위 versioning과 `budget B`를 enforced evaluation call로 연결한다. 논문 기본 benchmark 설정은 `B = 8`이다.

## 결과와 시사점
- benchmark task는 `MATH`, `GAIA`, `GPQA`, `SimpleQA`, `TAU-Bench Retail`로 구성된다.
- 결과는 task 특성에 따라 갈린다. reasoning-heavy task인 GPQA, MATH는 개선 폭이 작고, tool-use-heavy task인 GAIA, Retail, SimpleQA는 일관된 lift가 나타났다고 보고한다.
- 논문은 도구 사용형 task에서 대략 `8~9%` 수준 lift가 관찰된다고 정리한다.
- tracing 분석 결과, current optimizer는 대개 prompt modification에 과도하게 의존하고 modification diversity가 낮다고 한다.
- 또 optimizer instruction template에 따라 `variance-performance tradeoff`가 나타난다. 공격적인 instruction은 peak gain은 크지만 분산도 커지고 cross-task regression이 생길 수 있다.
- simpler target agent일수록 optimization headroom이 더 크다는 점도 case study에서 보인다고 적는다.

## Harness Engineering 관점
- harness engineering이 agent를 개선하는 대상일 뿐 아니라, 개선 과정을 측정하는 harness도 필요하다는 점을 잘 보여준다.
- agent optimizer를 비교하려면 target agent보다 먼저 evaluation harness를 제대로 설계해야 한다는 점을 보여 준다.
- versioned snapshot, structured trace, budgeted evaluator 같은 메커니즘은 self-improving agent 연구의 기본 인프라로 읽을 수 있다.

## 한계와 주의점
- 평가 프레임워크 논문이라 직접적인 product harness recipe는 제한적이다.
- budget을 evaluation call 수로만 정의해 token/API cost variance를 완전히 통제하지 못한다는 한계를 논문도 인정한다.
- target task는 유용하지만, production coding workflow 전체를 대변하진 않는다.
